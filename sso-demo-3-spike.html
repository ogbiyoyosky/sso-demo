Summary

Investigate moving authentication to the edge gateway and centralizing authorization in a single policy service, so application services hold no auth code. Concretely: Kong's OIDC plugin authenticates every request against Keycloak (which brokers to Microsoft Entra ID); Kong then forwards to an auth-service that evaluates a central policy and proxies allowed requests to the apps. Produce a recommendation (adopt / adopt-with-changes / reject) plus the open questions, risks, and a reference implementation.

Background & current state

We have validated three patterns in the sso-demo* PoCs. This spike evaluates the third for production adoption.

User identity is brokered: in-house staff sign in via Entra (granted the employee role); external users self-register with email + password (customer). All users persist in Keycloak's Postgres.

Goal of this spike (questions to answer)

  1. Is the community kong-oidc plugin viable on our Kong version (OSS 3.x), or do we need Kong Enterprise's OIDC / a different gateway?
  2. What is the correct issuer/hostname strategy so tokens validate while the gateway reaches Keycloak internally and Entra's redirect-URI constraints hold?
  3. Does centralizing authorization in one service give acceptable latency and an acceptable blast radius (it becomes a hot path / SPOF)?
  4. What belongs in central policy vs. what must stay in the app (object-level authZ)?
  5. What is the operational cost (HA, sessions/Redis, secret management, plugin maintenance)?

Decision boundary (the heart of the RFC): Kong answers "are you logged in?" (authentication). The auth-service answers "may you reach this host/path?" (authorization). Apps answer nothing — they trust headers set by the trusted edge.

AuthN = the first round-trip through Keycloak + session creation. AuthZ = the single POLICY lookup on the authenticated request. A customer hitting App Two authenticates successfully, then receives 403 — proving the two concerns are separated.

The issuer / hostname problem (must-solve)

OIDC clients validate the token iss, so the URL the browser uses and the URL the gateway uses server-side must yield the same issuer — while Entra only allows plain-http redirect URIs for the literal host localhost. lua-resty-openidc has no per-request URL rewrite hook, so we feed it a pre-built discovery table with split URLs:

   issuer / authorization_endpoint / end_session_endpoint  -->  http://localhost:8080   (browser-facing; iss matches)
   token_endpoint / userinfo_endpoint / jwks_uri           -->  http://keycloak:8080    (gateway-facing; reachable)
Open question for prod: behind real TLS + a public hostname this split collapses (one externally-resolvable issuer). Confirm the strategy for prod DNS so we don't ship a localhost-only hack. SPIKE TASK.

Code snippet — the central policy (the only "business logic" of authZ)

// auth-service: authZ is one declarative table + one check. Swap for OPA/Cedar later.
const POLICY = {
  'app-one.localhost': { appName: 'Customer Portal', requireRole: null },        // any authenticated user
  'app-two.localhost': { appName: 'Internal Tools',  requireRole: 'employee' },  // employees only
};

app.use((req, res) => {
  const host   = (req.headers.host || '').split(':')[0];
  const claims = JSON.parse(Buffer.from(req.headers['x-userinfo'], 'base64').toString()); // set by Kong
  const roles  = claims.roles || [];
  const rule   = POLICY[host];

  if (rule.requireRole && !roles.includes(rule.requireRole))      // <-- AUTHORIZATION
    return res.status(403).send(denyPage(rule, claims.email, roles));

  req.headers['x-auth-email'] = claims.email;                       // hand identity to the app
  req.headers['x-auth-roles'] = roles.join(',');
  proxy.web(req, res, { target: req.url.startsWith('/api') ? backend(host) : frontend(host) });
});

Spike tasks / investigation checklist

  • Stand up the reference stack (sso-demo-3) and complete a real Entra login end-to-end (employee + customer).
  • Confirm kong-oidc plugin viability on our pinned Kong OSS version; document the build (luarocks manifest workaround, 2.x→3.x port).
  • Validate the issuer split locally; design the production issuer/DNS/TLS strategy.
  • Measure added latency of the Kong→auth-service hop (p50/p95) under load.
  • Define HA topology for auth-service + Redis + Keycloak; estimate cost.
  • Decide policy engine: hardcoded table vs. OPA/Rego vs. Cedar; PoC the chosen one.
  • Define the trust boundary enforcement (network policy / mTLS so apps can't be called directly).
  • Map object-level authZ that must remain in apps; document the split contract (headers).
  • Logout completeness (RP-initiated / id_token_hint) — current plugin version is limited.
